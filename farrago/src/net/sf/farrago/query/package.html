Implements Farrago query processing.

Farrago Query Processing

This document provides an overview of how Farrago produces a JDBC ResultSet corresponding to the text of a SELECT statement. This should provide some appreciation for the term SQL engine, as well as a grasp of the most important interfaces and components in the system.

We'll use a simple, innocent-looking statement as an example: 


select *,42 as the_answer 
from sales.depts as d
where deptno = 10

Parsing

Our story begins when the prepareImpl method of {@link net.sf.farrago.jdbc.FarragoJdbcConnection} processes an SQL string by creating a new instance of {@link net.sf.farrago.parser.FarragoParser} and asking it to parse the statement. This is accomplished via an extension of {@link net.sf.saffron.sql.parser.SqlParser}, and the result (assuming no syntax error is thrown) is a tree of {@link net.sf.saffron.sql.SqlNode} objects:

Query Implementation Cache Lookup

FarragoJdbcConnection delegates the rest of the preparation to the prepareStmt method of {@link net.sf.farrago.db.FarragoDatabase}, which manages a shared cache of query plan implementations. For this discussion, we'll assume nothing is cached initially, so FarragoDatabase creates a new instance of {@link net.sf.farrago.query.FarragoPreparingStmt} to oversee the preparation process, and calls its validate() and implement() methods.

Validation

After parsing, the parse tree must be validated. prepareSql in {@link net.sf.saffron.oj.stmt.OJStatement} is called, this time taking the root SqlNode and an instance of {@link net.sf.saffron.sql.SqlValidator} (we supply a {@link net.sf.farrago.query.FarragoSqlValidator}). SqlValidator walks the parse tree performing validation tasks such as type inference, resolving identifiers, and expanding '*' in select lists; this results in a transformation of the parse tree (or a thrown exception if validation fails):
In addition, notice that the validator has augmented the parse tree with a scope tree (in this case a simple two-level linear tree since there are no joins). The top-level scope represents the columns produced by the SELECT list; the bottom scope represents the columns to be read from table SALES.DEPT. Note that each SqlIdentifier node in the validated tree is either
  • a table reference (e.g. SALES.DEPT)
  • a reference to a column produced by a lower-level scope (e.g. D.DEPTNO)
  • a new column being produced at this scope (e.g. THE_ANSWER)
In the presence of subqueries, there are other possibilities such as a reference to a column produced by an enclosing scope.

Translation to Relational Algebra

The next step is to convert the validated SQL parse tree into a relational algebra expression. The dirty work is all performed by {@link net.sf.saffron.sql2rel.SqlToRelConverter}, relying on the validator's scope and type information. The result is again a tree, but this time of instances of {@link net.sf.saffron.rel.SaffronRel}. The target set of relational expressions is constrained to the following abstract operators:
  • {@link net.sf.saffron.rel.TableAccessRel}
  • {@link net.sf.saffron.rel.TableModificationRel}
  • {@link net.sf.saffron.rel.ProjectRel}
  • {@link net.sf.saffron.rel.FilterRel}
  • {@link net.sf.saffron.rel.JoinRel}
  • {@link net.sf.saffron.rel.CorrelatorRel}
  • {@link net.sf.saffron.rel.AggregateRel}
  • {@link net.sf.saffron.rel.UnionRel}
  • {@link net.sf.saffron.rel.IntersectRel}
  • {@link net.sf.saffron.rel.MinusRel}
  • {@link net.sf.saffron.rel.OneRowRel}
The EXPLAIN PLAN statement can be used to see the translated tree for our example: 

explain plan without implementation for
select *,42 as the_answer 
from sales.depts as d
where deptno = 10
This yields: 

ProjectRel(DEPTNO=[$0], NAME=[$1], THE_ANSWER=[42])
  FilterRel(condition=[=($0,10):BOOLEAN])
    FennelIndexScanRel(table=[SALES.DEPTS], projection=[*], index=[SYS$CONSTRAINT_INDEX$DEPTS$SYS$PRIMARY_KEY],preserveOrder=[false])
Here's a graphical representation:
The three-dimensional boxes are relational algebra nodes (instances of SaffronRel); the two-dimensional boxes are validated row expressions represented by the {@link net.sf.saffron.rex.RexNode} object model. Besides children and row expressions, relational expressions may have additional attributes; e.g. ProjectRel has an array of names to be assigned to the columns it produces, and FennelIndexScanRel has a reference to the table to be scanned. Within a row expression, a {@link net.sf.saffron.rex.RexInputRef} may refer to the output of a child relational expression via variables with names like $N (hence the $0 in the ProjectRel's expression refers to something different from the $0 in the FilterRel's condition expression, since the expression is relative to the containing SaffronRel).

Note for Nit-Pickers: the EXPLAIN PLAN output above is not completely abstract, since TableAccessRel has already been replaced with FennelIndexScanRel during translation.

Optimization

Query optimization transforms the abstract relational algebra expression described above into an equivalent expression consisting only of operators for which physical implementations are available. For an overview of the process, see {@link net.sf.saffron.opt the Saffron optimizer docs}. To tailor the optimization process for our requirements, {@link net.sf.farrago.query.FarragoPlanner} subclasses {@link net.sf.saffron.opt.VolcanoPlanner} to define Farrago-specific rules, relational expressions, and calling conventions.

Farrago constrains the optimizer to the following calling conventions:

  • FENNEL: data processing implemented by Fennel TupleStreams (TODO: link)
  • ITERATOR: data processing implemented in Java by classes which implement the Iterator streaming interface
  • RESULT_SET: data returned via JDBC result sets, either as the output of a Farrago query, or as leaf-level input from foreign databases (NOTE: this isn't implemented yet)
To see the output of the optimizer, we can use EXPLAIN PLAN again: 

explain plan for
select *,42 as the_answer 
from sales.depts as d
where deptno = 10
(This time WITHOUT IMPLEMENTATION was left out, and the default is WITH IMPLEMENTATION, which is what we want.) This yields: 

IterCalcRel(DEPTNO=[$0], NAME=[$1], THE_ANSWER=[42])
  FennelToIteratorConverter
    FennelIndexSearchRel(table=[SALES.DEPTS], projection=[*], index=[SYS$CONSTRAINT_INDEX$DEPTS$SYS$PRIMARY_KEY], uniqueKey=[true], preserveOrder=[false],outer=[false],inputKeyProj=[*],inputJoinProj=[[]])
      IteratorToFennelConverter
        IterCalcRel(field#0=[CAST(10):INTEGER])
          IterOneRowRel
which corresponds to the following relational expression tree:
Wow, what happened? The optimizer found that the table's clustered index could satisfy the WHERE clause, and accordingly transformed the query using a {@link net.sf.farrago.query.FennelIndexSearchRel} for the table access. (It knew how to do this because FarragoPlanner told it about {@link net.sf.farrago.query.FennelScanToSearchRule}.) What's interesting is that this search is sandwiched between Java iterators. The lowest level ({@link net.sf.saffron.oj.rel.IterOneRowRel}) produces a single row with no columns; the next level up ({@link net.sf.saffron.oj.rel.IterCalcRel}) produces the constant DEPTNO value (10) which we're searching for. (This can remain anonymous as field#0 since Fennel accesses data by tuple position, not value.) Now, the optimizer wants to feed this into the Fennel search as the input key, but the search has FENNEL calling convention, while IterCalcRel has ITERATOR calling convention. So, a {@link net.sf.saffron.rel.convert.ConverterRel} is required; in this case, {@link net.sf.farrago.query.IteratorToFennelConverter}.

But wait, there's more! The FennelIndexSearchRel is going to produce matching rows in FENNEL representation, but in order to compute the select list and return results via JDBC, we need another converter: {@link net.sf.farrago.query.FennelToIteratorConverter}. The top-level IterCalcRel then implements the original abstract ProjectRel.

Physical Implementation

Once the optimizer has decided on the best implementation, it's time to generate corresponding executable code. This is the job of {@link net.sf.saffron.opt.RelImplementor}, which walks the optimized SaffronRel tree calling implement on each expression, producing the parse tree representation of a Java class. From this, the query processor generates the Java class definition as text, saves it to a file, and then invokes the Java compiler and loads the resulting classfile. Yeesh, that's ugly! And that's after I replaced a huge XML string in the middle with BIG-FAT-XML-STRING. How can we interpret this in the context of the implementation plan?
  • The static inner classes represent the runtime types of the output rows and columns for each relational expression. For example, Ojp_2 has fields DEPTNO and NAME, and represents the results of searching the table. VARCHAR_128_ISO_8859_1 is the non-primitive type for the NAME column, embodying additional type information such as its precision and character set.
  • Method execute() is the top-level Iterator producing the result set. Its body is, well, a true farrago of anonymous inner classes. The details are beyond the scope of this document, but we'll cover a few of the more important points.
  • The execute() method gets passed an instance of FarragoRuntimeContext, which it uses for services such as communicating with Fennel.
  • The first calcNext() method corresponds to the lowest-level IterCalcRel. It computes the constant DEPTNO search key (10). Note that the call to java.util.Collections.singletonList came from IterOneRowRel.
  • The second calcNext() method corresponds to the top-level IterCalcRel. It passes through DEPTNO and NAME and tacks on THE_ANSWER (42).
  • The unmarshalTuple() method contains code generated to read data from the Fennel TupleStream using java.nio.
  • The Fennel portion of the plan is represented by BIG-FAT-XML-STRING. On execution, this will be compiled into a graph of Fennel TupleStreams which do work such as BTree access. More on this below.

Fennel Implementation

Let's look more deeply into the Fennel portion of the plan. When RelImplementor reaches a FennelToIteratorConverter in its traversal of the optimized tree, the corresponding implement method performs a sub-traversal of the connected subtree of nodes implementing the {@link net.sf.farrago.query.FennelRel} interface. Rather than generating Java code, these nodes generate (via the FennelRel.toTupleStreamDef method) a tree of {@link net.sf.farrago.fem.fennel.FemTupleStreamDef} instances specifying the Fennel portion of the plan. FennelRelToIteratorConverter exports this tree as an XMI string which is then "burned into" the generated Java code. Here's the XML for our example:

Execution

Whew. Now we actually want to execute the generated code. This part is easy. We use reflection to invoke the generated execute() method, which returns an Iterator instance, and pass that to a new instance of {@link net.sf.farrago.runtime.FarragoIteratorResultSet}, which produces JDBC result data and metadata, as well as taking care of cleanup when the result set is closed.

Palliatives

If you have read through this entire doc, you are probably wondering how such a heavyweight procedure is ever going to provide decent performance in the context of the kind of rapid-fire statement execution used in transactional applications. Here are some planned improvements:
  • When the optimizer estimates that query execution will be very brief, we won't bother compiling the generated Java code. Instead, we'll interpret it using DynamicJava. (For now, we always use DynamicJava.)
  • The generated Java code is going to look very different once the code generator has been overhauled for full SQL expression evaluation. The current implementation is just "proof of concept" quality.
  • To improve performance of one-shot queries, the Fennel plan could be left in object format and passed to Fennel directly rather than having to export it as XMI and later import it on execution. For re-executed statements, we already keep a cache of reusable Fennel TupleStream trees rather than reconstructing one at the start of each execution.
  • Wherever Java overhead turns out to be too high (perhaps for example, generating the search key for use by Fennel in the example query), Fennel will be enhanced to do the job instead (e.g. by allowing constant tuple data to be represented directly in the stream definition). A project is already in progress to implement all builtin SQL expressions via C++ (the Fennel calculator component). However, the need to call extensions written in Java or other languages will always be present, which is why we started with the most general case first.
  • Applications using prepared SQL statements or stored procedures will pay the query preparation cost only once during query preparation; repeated executions will invoke the generated Java class directly.
  • For less sophisticated applications, we'll implement transparent query implementation caching with pattern matching for not-quite-identical queries (e.g. different only by the literal values used in an INSERT). Query plan caching for identical queries is already implemented by FarragoDatabase (after some match-improving normalization such as fully qualifying identifiers is performed by validation).
