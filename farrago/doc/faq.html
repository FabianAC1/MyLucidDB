Does the world really need another open-source DBMS?

Perhaps surprisingly, yes. Most of the offerings I've seen are fine for running a website but inadequate for enterprise use. In addition, the most popular open-source systems are burdened with legacy code and architecture. There are many ways to go wrong, especially if the goal is something as banal as a quick buck, mindshare, or a dumping ground for an unsuccessful commercial product. The basics of relational database technology are mature, well-known, and non-proprietary; despite its name, Farrago is all about taking the time to create a maintainable, high-quality implementation that can eventually replace even the most heavyweight closed-source products (OK, that and achieving world peace). Novelty is a non-goal.

Isn't Java slow? Doesn't that mean Farrago can't possibly compete with systems implemented entirely in C/C++?

Careful attention has been paid to keep Java out of the most important data processing paths. Where Java is used (mostly at the metadata and optimization level, with an auxiliary role in execution), its performance is acceptable.

Under what license is Farrago distributed?

The Farrago code which puts everything together is licensed under the GNU LGPL (GPL would be preferable, but is incompatible with usage of Java). MDR components are licensed under the Sun Public License. MonRG is under the Common Public License. JavaCC is under the BSD License.

Some older Saffron documentation describes many deviations from standard SQL (e.g. null semantics, Java type system dependencies, aggregation). Does this mean Farrago SQL is non-standard also?

No. The core of Saffron is an extensible optimizer which is agnostic about these issues. Farrago supplies its own SQL-conformant optimization rules when it instantiates the optimizer. The other parts of Saffron (such as the Java precompiler) aren't used for SQL processing.

Are you insane? Why are you even attempting this?

John Sichi's personal answer: Maybe. Because. Ok, at least three reasons:
  • The giants are out there. They're not windmills. Someone's got to take them on.
  • I have built significant portions of four different DBMS products in my for-hire career. Each time, I ended up having to walk away from a lot of code, usually due to changes in company direction. I need closure.
  • It seems that every few years someone gets an idea for a new DBMS which is going to solve some niche problem, thus avoiding direct competition with the big boys. (I've been involved with a couple of these.) What happens is that 90% of the effort gets burnt on building all the infrastructure required for a generic DBMS, and only 10% on the proprietary solution. This is counter-productive. So, even if Farrago never achieves its most ambitious goals, its architecture should still make it a useful platform for this kind of effort.

Why a jackalope for the mascot?

The wily jackalope combines the speed of a jackrabbit with the fluid grace and complex headgear of an antelope, which seems appropriate for a hybrid Java/C++ system. In addition, the jackalope symbolizes a vision beyond the mundane reality seen by others (and is also reputed to be surprisingly savage when cornered).
